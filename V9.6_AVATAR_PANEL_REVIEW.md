# Avatar System Panel Review — V9.6 follow-up

**Question on the table:** expand the curated avatar library, and decide whether to allow user-uploaded profile images. If we allow uploads, do we need manual review?

**Today's state:**
- Curated library of preset avatars (alien, ghost, ufo, etc. — `<AvatarSelector>` component)
- Stored as `avatar_url` strings on `profiles.avatar_url`, pointing to either a built-in slug or an external URL
- No upload flow; no moderation pipeline
- No image processing (crop / resize / EXIF strip)

**Chase's note:** he has more icons ready to add, and is open to letting users upload their own with manual review.

---

## The panel

### 1. Mira Chen — Mass-market UX (consumer apps, ex-Spotify)

> The curated library is the right starting point for an indie consumer app. Spotify, Notion, Linear, Discord, Slack — all of them give you a built-in choice plus optional custom upload. The "I want to be myself" people upload, the "I just want to ship" people pick from the grid. Don't make either group feel second-class.

**Top recommendation:** ship a hybrid system. Keep the curated grid as the default and primary choice (it's on-brand, it's free, it has zero abuse risk), then add a "+Custom" tile at the start of the grid that opens an upload flow. New users default into the curated grid so onboarding stays fast.

**Second:** when expanding the curated set, organize by theme (cryptids · UFO · ghosts · researchers · symbols), not as one long scroll. ~24-40 icons total is the sweet spot — enough to feel personal, few enough to scan.

---

### 2. Imani Brooks — Trust & Safety / content moderation (ex-DuckDuckGo)

> Custom avatar uploads are the single most common abuse vector on small consumer apps. Top failure modes I've seen, in order of frequency:
>
> 1. NSFW / explicit imagery
> 2. CSAM (catastrophic — instant federal exposure)
> 3. Hate symbols, swastikas, slurs as text-in-image
> 4. Trademark infringement (corporate logos)
> 5. Public-figure impersonation (using a celebrity photo)
> 6. Doxxing / harassment (targeting another user)
>
> A manual-only review queue won't scale past a few hundred uploads/day and creates a moderator-fatigue / decision-quality cliff. Manual-only ALSO has legal exposure problems — if you have a manual queue and miss CSAM, that's worse than not having a queue at all because you "should have caught it."

**Top recommendation:** **automated first-line scan + manual review queue for flagged items.** Specifically:

1. **First line: automated moderation** — every upload passes through a vision API (AWS Rekognition Moderation Labels, Google Cloud Vision SafeSearch, or Hive Moderation) that returns confidence scores for: explicit-nudity, violence, hate-symbols, weapons, CSAM-like-features, alcohol/drug-context. Rekognition's "Detect Moderation Labels" is ~$1 per 1000 images — trivial.
2. **Auto-reject** on high-confidence bad content. Show user a friendly error: "We weren't able to use that image. Please pick something else or use one of our preset avatars."
3. **Auto-approve** on clear-positive content (high confidence not-bad).
4. **Manual queue** for the middle band (low confidence either way, ~5-10% of uploads). Admin reviews via the existing `/admin/media-review` page (you have one).
5. **CSAM**: any hit on Rekognition's CSAM-related labels → auto-reject + send the user account to admin queue + log to a tamper-evident audit table. Don't try to make decisions here — escalate.

**Second recommendation:** require a **safe default** while moderation is pending. If a custom upload is queued, the user's avatar shows the curated default they previously had (or a generated placeholder). They don't get to walk around with an unreviewed image visible to other users.

---

### 3. Sam Park — Privacy & data minimization (WCAG 2.2 / GDPR)

> Two concerns most consumer apps miss:
>
> 1. **EXIF metadata.** Phone photos carry GPS coordinates by default. If a user uploads a selfie they took at home, you've now stored their home address as metadata next to their profile. Strip on upload.
> 2. **Right to be forgotten.** When a user deletes their account, their uploaded image needs to actually be deleted from storage too — not just the DB row. GDPR Article 17.

**Top recommendation:** strip all EXIF on upload. Process server-side with `sharp` or similar — re-encode the image, drop metadata, resize to a max edge length (256x256 is plenty for an avatar), output AVIF or WebP for size.

**Second:** when an account is deleted, queue the avatar file for deletion from Supabase Storage. Don't rely on cascade deletes — use an explicit cleanup job in the deletion flow.

**Third:** **alt text.** A custom-uploaded avatar should let the user provide alt text. Curated avatars come with alt baked in ("UFO icon", "Ghost icon"). Custom uploads need user input or default to "@username avatar."

---

### 4. Devon Park — Design systems lead

> The curated library is currently a closed set hardcoded in `AvatarSelector`. To expand, the question is: do we keep growing the hardcoded list (each addition = code deploy), or move to a data-driven approach (DB table of curated avatar options)?

**Top recommendation:** move curated avatars to a `curated_avatars` table now while it's small. Schema: `id, slug, name, category (cryptid/ufo/ghost/researcher/symbol), image_url, sort_order, is_active`. AvatarSelector fetches from `/api/avatars/curated`. Adding new avatars becomes a SQL insert + image upload, no deploy required. Future: lets you A/B test avatar sets, retire underused ones, or surface seasonal additions.

**Second:** standardize all avatar dimensions to a single source-of-truth size (recommend 256x256 PNG/WebP). Pre-render `@1x` (128px) and `@2x` (256px) variants for the rendered UI. Inconsistent dimensions are how you end up with one avatar looking pixelated next to another.

---

### 5. Jordan Reyes — iOS / mobile specialist

> Mobile users will overwhelmingly want to upload from their camera roll, not pick from a grid. The native iOS file picker is good — use `<input type="file" accept="image/*">` and let iOS handle the camera-roll vs camera-shot toggle. On Android same pattern works.
>
> Watch out: iOS Safari sometimes uploads HEIC files (Apple's default photo format), which not every server image library handles. Convert HEIC → JPEG/WebP server-side or reject HEIC at the client with a clear error.

**Top recommendation:** mobile-first upload flow with in-app crop. Library: `react-easy-crop` or similar. Force a square 1:1 crop before upload so we don't have to deal with rectangular rabbit holes. On submit, send the cropped result (already a Blob) to the upload API, not the original file.

**Second:** PWA install on iOS limits some camera APIs. Test the upload flow inside the installed PWA before considering this shipped — don't just test in mobile Safari.

---

### 6. Rae Kowalski — Retention / growth

> Custom avatars are sneakily great for retention. Users who customize their profile early are 2-3x more likely to come back day-2 (every consumer-app retention dataset I've seen confirms this). The act of personalizing creates a feeling of ownership. So this isn't just a feature — it's a retention lever.

**Top recommendation:** prompt the avatar choice during onboarding, not later. After a new user signs up, route them through a quick "pick your researcher avatar" step. Curated grid first; the "+Custom" tile is the secondary CTA. Don't gate it as required — let people skip with a default.

**Second:** show the user's avatar back to them in many places (Today header, save confirmations, streak chip, comment threads when those exist). The more they see "their" avatar, the more it becomes part of identity.

## Synthesized plan — 4 phases

### Phase 1 — Curated library expansion (ship first)

**Scope:** add Chase's new icons to the existing curated library, organized by theme.

1. Create `curated_avatars` Supabase table with the schema above.
2. Migrate existing hardcoded avatars into the table.
3. Update `<AvatarSelector>` to fetch from `/api/avatars/curated` and group by `category`.
4. Add a category tab strip inside the picker: **Cryptids · UFO · Ghosts · Researchers · Symbols** (or whatever themes the new icons fit).
5. Pre-render at 256x256 WebP, store in Supabase Storage under a public bucket.

**Estimated scope:** ~1 day. New table + migration + ~150 LOC.

### Phase 2 — Custom upload (full implementation, automated moderation primary)

1. **Client:** add `+Custom` tile at the start of the avatar grid. Opens a square-crop UI (react-easy-crop). On submit, sends a cropped Blob.
2. **Server:** new `POST /api/user/avatar/upload` endpoint:
   - Accept multipart/form-data with the cropped image
   - Reject HEIC client-side; if it gets through, convert via sharp
   - Re-encode to 256x256 WebP, strip EXIF, max 200KB
   - Send to AWS Rekognition Moderation API
   - On clear-pass: write to Supabase Storage, update `profiles.avatar_url`, return success
   - On clear-fail: reject with specific user-friendly error
   - On uncertain: store with `is_pending_review = true`, keep user's previous avatar visible to others, send to admin queue
3. **Admin queue:** extend existing `/admin/media-review` to surface pending avatars. Admin clicks Approve / Reject / Reject + warn user.
4. **Storage:** Supabase Storage bucket `avatars-user`, path `{user_id}/{uuid}.webp`. Public read, authenticated write.
5. **Deletion:** on account deletion, queue avatar file removal.

**Estimated scope:** ~3-4 days. New endpoint + Rekognition integration + crop UI + admin review extension.

### Phase 3 — Onboarding integration

1. After signup completion, route new users to a "Pick your avatar" step.
2. Curated grid first (organized by theme as in Phase 1).
3. `+Custom` tile as secondary CTA.
4. Skip button → assigns a default deterministic avatar (e.g., one of the cryptids based on user_id hash).

**Estimated scope:** ~1 day.

### Phase 4 — Polish

1. Alt-text input for custom avatars (defaults to "@username avatar")
2. EXIF strip verified server-side
3. Replacement frequency limit: 5 custom uploads per 24h to prevent abuse
4. Admin moderation log + dashboard for tracking decisions

**Estimated scope:** ~1-2 days.

## Direct answer to Chase's questions

**Q: Should we allow user uploads?**
Yes — panel consensus. Retention upside is real, the abuse risk is manageable with automated moderation, and it's expected functionality at this point in a consumer app.

**Q: Do we need manual review?**
Hybrid is best. Pure manual doesn't scale past hundreds/day and creates legal exposure (what you don't catch is worse than what you don't try to catch). Pure automated misses edge cases. The "automated first-line + manual queue for flagged" pattern handles 95%+ of uploads automatically and gives you human eyes on the genuinely ambiguous 5%.

**Q: What does best practice look like?**
- AWS Rekognition Moderation Labels (~$1 per 1000) or Google Cloud Vision SafeSearch — pick one
- Server-side image processing (sharp) for resize + format convert + EXIF strip
- 256x256 WebP output, public-read storage
- Admin queue for low-confidence cases, surfaced in `/admin/media-review`
- Safe-default behavior while pending (show curated default to other users)
- Right-to-be-forgotten: cascade delete on account close
- Replacement frequency limits on the upload endpoint
