## 2. Issue B — FindingCard ordering is data-first, not prose-first

Founder pass after V11.18.18 SQL went live: "the cards still feel like
study results. The numbers come first and the prose comes after — it
should be the other way around." Per the Gaia rules in
`docs/GAIA_FINDING_COPY.md` §2 Rule 1 ("Lead with the qualitative
finding, not the math"), the interpretive sentence should be the
visual anchor on every variant, not a footnote beneath the hero %.

### 2.1 Fix — prose-led layout across all 3 variants

`src/components/patterns/FindingCard.tsx` — Sprint 1G rewrite of the
body order for every variant.

**Before** (today_card example):

```
eyebrow → headline → hero % → per-family bars → denominator → prose → rep-report slab → footer
```

**After** (today_card):

```
eyebrow → headline → INTERPRETIVE PROSE → [See the numbers] (collapsed by default) → rep-report slab → footer
```

The new `NumbersDisclosure` component holds the hero %, per-family
bars, and denominator behind an expandable "See the numbers"
button. Defaults closed; per-render only (Today swipe deck remounts
on every cycle so the disclosure resets, matching the "data never
leads" register the panel ratified).

Per variant:

| Variant | Prose location | Data zone |
|---|---|---|
| `rail` | Card body, line-clamped to 5 lines on desktop grid; full on mobile rail | Suppressed entirely — no hero, no bars, no denominator on the rail. The Gaia register asks for a small documentary observation, not a study result. |
| `grid` (full /lab/patterns page) | Below the headline | Behind `NumbersDisclosure`; researchers click to open |
| `today_card` | Body line — 15/16px, bumped legibility | Behind `NumbersDisclosure`; preview rep-report excerpt slab + overlay still render between |

Footer (Paradocs Archive + N accounts + Read more →) is unchanged on
every variant — it's the citation, not data leading the read.

### 2.2 New `desktopGrid` prop on the `rail` variant

V11.18.19 — Sprint 1G. The `rail` variant now accepts a `desktopGrid`
boolean. When true:

- Width: `w-full` (fills the grid cell) instead of fixed
  `w-[300px]/sm:w-[324px]` + `shrink-0`.
- Max height: ~280px, so the desktop grid cards are scannable rather
  than tall.
- Prose: line-clamped to 5 lines.

This single prop lets PatternsRail (see §4) render the same `rail`
card body in either the mobile horizontal scroll OR the desktop
2-column grid. Default false — every other consumer (e.g. anywhere
else that renders the rail card) is unaffected.

## 3. Issue C — Geographic Cluster substance zone still empty

Founder screenshot of the California UFO cluster showed an empty
substance zone despite V11.18.16's `fallbackContext` backstop. Three
hardenings this pass.

### 3.1 Diagnostic surface

`src/pages/api/discover/clusters.ts` — `loadClusterReports` now
**always** emits a single structured log line on every invocation,
per the founder's spec:

```
[loadClusterReports] { cluster: 'geo-California-ufos-aliens',
                       category: 'ufos_aliens', state: 'California',
                       ids_supplied: 24, in_query: 22, fallback: 0,
                       total: 3 }
```

Lands in Vercel logs on every cluster build so the operator can
audit the backstop fired on the success path, not only when it
failed.

### 3.2 Fallback always runs when IN-query < 3

Previously the fallback was conditional on
`out.length < 3 && fallbackContext && fallbackContext.category`. The
condition was correct in principle, but the function signature now
**requires** `fallbackContext: { category, state?, clusterIdForLog? }`
— callers cannot accidentally omit the category. When the IN-query
returns fewer than 3 rows the fallback fires unconditionally; when it
can't fire (category is somehow falsy at runtime), the helper emits
a warning instead of silently rendering nothing.

### 3.3 Fallback window widened 7d → 30d

The cluster itself is assembled from a 7-day window. The previous
fallback was also 7-day, so if the past 7 days were thin enough that
the linked-ID resolution missed rows, the fallback was likely to also
return 0. Bumped to 30 days. The cluster's *summary* (count, baseline
ratio) is still computed off the 7-day window — only the substance
zone's "in this cluster" list reaches a little wider when needed.

### 3.4 ClusteringCard graceful empty state

`src/components/discover/ClusteringCard.tsx` — when both the
IN-query AND the 30-day fallback come up empty (rare — corpus
genuinely has 0 matching reports), the substance zone renders a
graceful placeholder instead of collapsing to nothing:

```
IN THIS CLUSTER
Reports for this cluster will appear here as the archive indexes them.
Browse all <Category> →
```

The "Browse all <Category> →" link routes to `/explore?category=…`
so the card is never a dead-end. Documentary register, no apology.

### 3.5 Files

- `src/pages/api/discover/clusters.ts` — `loadClusterReports`
  rewritten (mandatory `fallbackContext`, 30-day window,
  single-line `[loadClusterReports]` diagnostic, cluster id passed
  through for log correlation). Both call sites updated.
- `src/components/discover/ClusteringCard.tsx` — `reps.length === 0`
  now renders placeholder + "Browse all …" link; diagnostic
  message updated to "both IN-query and 30-day fallback returned 0
  rows".

## 4. Issue D — PatternsRail clunky on desktop

Founder feedback: the horizontal-scroll rail of 3 full Finding cards
reads as clunky at desktop widths. Desktop users have a wide canvas
and would rather scan than scroll-past.

### 4.1 Fix — 2-column grid at lg breakpoint

`src/components/lab/PatternsRail.tsx`:

- Fetch limit bumped 5 → 6 (so the desktop grid renders 3 even rows
  of 2 cards).
- Mobile (`< lg`): unchanged. Horizontal-scroll rail with
  `snap-x snap-mandatory`. The horizontal grammar is the right
  grammar for a discovery surface on a narrow viewport.
- Desktop (`≥ lg / 1024px`): the rail switches to a
  `lg:grid lg:grid-cols-2 gap-3` layout. Each cell renders the
  `rail` FindingCard variant with `desktopGrid` true, which produces
  the V11.18.19 compact prose-only card (max-h ~280px, line-clamped
  to 5 lines, no data zone).
- Header ("Patterns from the archive" / "See all patterns →") is
  unchanged on both viewports.

The grid sits inside lab.tsx's existing `max-w-3xl` container
(~768px), so each cell is ~340px wide which mirrors the mobile rail
card width. No layout regressions.

### 4.2 Files

- `src/components/lab/PatternsRail.tsx` — desktop grid block added
  beneath the existing horizontal scroll block, gated with
  `hidden lg:grid` / `lg:hidden` so exactly one renders per
  viewport.
- `src/components/patterns/FindingCard.tsx` — `desktopGrid` prop on
  the `rail` variant (see §2.2).

## 9. Open questions for founder

1. **"See the numbers" disclosure default state.** Set to closed on
   every render. Tap to open. Should the disclosure remember its
   open state for the session (e.g. once a researcher opens it on
   one card, future card visits open it pre-expanded too)? Easy
   add — localStorage-backed — but the panel landed on "no, the
   default should always be prose-first" because the moment we
   remember the open state we're back to data-first for repeat
   visitors. Flag if this should change.
2. **Cluster fallback time window.** 30 days might still be too
   narrow for slow-trickle categories (e.g. cryptid clusters). If
   the founder sees the placeholder rendering on a cluster that
   *should* have reports, the next ratchet is 90 days. Hold for
   QA evidence.
3. **PatternsRail desktop grid count.** Currently 6 (3 rows × 2
   cols). If the catalogue grows past 12 Findings the grid is the
   place to scale row count first. The standalone /lab/patterns
   grid remains the full surface; the rail is a sampler.
4. **"View from above" vs "out-of-body view"** for the
   obe_observer_from_above headline. The Gaia register prefers "view
   from above" (concrete, no jargon) over "out-of-body view" (closer
   to the catalogue term but reads slightly clinical). Picked "view
   from above" for the headline; the interpretive_sentence already
   uses "rise out of their bodies" so the OBE phrasing is still
   present one layer in. Flag if "out-of-body view" reads better
   to the founder.
