## Architectural diagnosis

### The race that V11.18.14 fixed for Finding + OnThisDate

`/discover` uses a TikTok-grammar vertical swipe component that calculates snap-points from the initial `items[]` array. When a special card was injected AFTER first paint (via the async `fetchSpecialCards()` → `pendingSpecialCards.current.push(...)` → `injectPendingSpecialCards()` → `setItems(prev => splice)` path), the snap-points were already calculated against the pre-injection array. The forward swipe blew past the special card's index, which only materialized on swipe-back when the swipe component re-evaluated layout.

V11.18.14 closed the loop for Finding + OnThisDate: SSR pre-fetched them in `getServerSideProps`, then `loadFeed(0)` spliced them into the initial array BEFORE the first `setItems()` call. Snap-points are now correct on the first forward swipe.

### Why cluster wasn't fixed in V11.18.14

The cluster path was still:

```typescript
// fetchSpecialCards() — runs AFTER first paint
fetch('/api/discover/clusters')
  .then(...)
  .then(data => {
    var clusterCard = Object.assign({}, available[pickIdx], { item_type: 'cluster' as const })
    pendingSpecialCards.current.push({ card: clusterCard, position: 8 })
  })
```

Result: cluster invisible on first forward swipe; only surfaces on swipe-back or after a page load.

### V11.18.20 fix

Cluster joins the SSR-bake pipeline. Same shape as the Finding + OnThisDate pattern; the cluster card now lands at idx 8 in the initial `items[]` array.

## Implementation — per step

### Step 1 — `DiscoverPageProps` extended

```typescript
interface DiscoverPageProps {
  initialFinding: FindingFeedItem | null
  initialOnThisDate: OnThisDateData | null
  initialCluster: ClusterCardData | null   // NEW
}
```

`var initialCluster = props.initialCluster || null` added to the function body.

### Step 2 — `ssrClusterDelivered` ref

```typescript
var ssrFindingDelivered    = useRef<boolean>(initialFinding != null)
var ssrOnThisDateDelivered = useRef<boolean>(initialOnThisDate != null)
var ssrClusterDelivered    = useRef<boolean>(initialCluster != null)   // NEW
```

Synchronous useRef initializer (V11.18.11 pattern) so the flag is set before the first `loadFeed(0)` fires.

### Step 3 — Splice cluster into initial items array (in `loadFeed(0)`)

```typescript
if (ssrOnThisDateDelivered.current && initialOnThisDate && !categoryFilter) {
  bakeSpecials.push({ card: initialOnThisDate, position: 1 })
}
if (ssrFindingDelivered.current && initialFinding && !categoryFilter) {
  bakeSpecials.push({ card: initialFinding, position: 4 })
}
// V11.18.20 — NEW
if (ssrClusterDelivered.current && initialCluster && !categoryFilter) {
  bakeSpecials.push({ card: initialCluster, position: 8 })
}
bakeSpecials.sort(function (a, b) { return b.position - a.position })   // DESC
bakeSpecials.forEach(function (s) {
  var pos = Math.min(s.position, merged.length)
  merged.splice(pos, 0, s.card)
})
```

Position 8 matches the legacy client-side push so the cluster occupies the same swipe slot the founder is used to.

Splice is DESC-sorted so tail-first inserts don't perturb earlier target indices in the same pass.

### Step 4 — Skip client-side cluster fetch when SSR delivered

```typescript
if (!categoryFilter && ssrClusterDelivered.current) {
  console.log('[Discover/Cluster] SSR-delivered — skipping client clusters fetch')
} else if (!categoryFilter) {
  // existing client fetch + pendingSpecialCards.current.push at position 8
}
```

Safety net unchanged: when SSR returns null (no clusters today, env not configured, fetch threw), `ssrClusterDelivered.current` stays false and the client-side fetch fires as the fallback — same shape as the Finding + OnThisDate safety nets.

### Step 5 — Reset ref on lens / category / pull-to-refresh

Mirroring the existing ssrFindingDelivered + ssrOnThisDateDelivered resets:

- `handleLensChange(...)` → `ssrClusterDelivered.current = false`
- `handleCategoryChange(...)` → `ssrClusterDelivered.current = false`
- Pull-to-refresh branch → `ssrClusterDelivered.current = false`

After reset, the next `loadFeed(0)` skips the SSR-bake splice (because the ref is false) and the client fetch fires as the path of record — same contract as Finding + OnThisDate.

### Step 6 — `getServerSideProps` — parallel cluster fetch

```typescript
var initialCluster: ClusterCardData | null = null
try {
  var clusterRes: any = await fetch(base + '/api/discover/clusters')
  if (clusterRes && clusterRes.ok) {
    var clusterData = await clusterRes.json()
    if (clusterData?.clusters?.length > 0) {
      // Deterministic UTC day-of-year pick
      var nowDt = new Date()
      var startUtc2 = Date.UTC(nowDt.getUTCFullYear(), 0, 1)
      var nowUtc2 = Date.UTC(nowDt.getUTCFullYear(), nowDt.getUTCMonth(), nowDt.getUTCDate())
      var doyC = Math.floor((nowUtc2 - startUtc2) / 86400000) + 1
      var pickIdxC = doyC % clusterData.clusters.length
      var picked = clusterData.clusters[pickIdxC]
      initialCluster = Object.assign({}, picked, { item_type: 'cluster' as const })
    }
  }
} catch (_e) { initialCluster = null }
```

Notes:
- Uses internal HTTP call to `/api/discover/clusters` so server + client see the same candidate pool (avoids duplicating the geographic/temporal/category-trend scoring + Haiku body generation logic).
- Day-of-year pick replaces the legacy session-seeded pick from `fetchSpecialCards`. The trade-off: less novelty per session, but a stable first-paint identity ("today's cluster is X") consistent with how Finding rotates. This matches what the founder is optimizing for.
- `base` URL hoisted out of the OTD block so the cluster fetch can reuse it.
- Same defensive try/catch + null fallback as Finding + OnThisDate. Return shape: `{ initialFinding, initialOnThisDate, initialCluster }`.

### Step 7 — Diagnostic log updated

`[Discover/SSR-finding]` log gained `hasInitialCluster`, `clusterId`, `clusterType`.

`[Discover/SSR-bake] items_bake_complete` already had `cluster_idx` (V11.18.19 baseline) — it now reports a value >= 0 when SSR delivered.

## Open questions

1. **Day-of-year vs session-seed pick.** Legacy client path used `sessionSeed.current % clusters.length` so each session saw a different cluster. SSR-bake uses day-of-year for stable first-paint identity. If the founder later wants novelty back: add a server-side rotation across the top-3 clusters per UTC 8-hour window, or restore session-seed novelty on the client-fetch safety path while keeping day-of-year for SSR.
2. **Cluster freshness vs SSR cache.** Clusters refresh as ingestion lands new approved reports; a 60s edge cache is tight enough that this is invisible to the user. If the cluster pool ever shrinks to a single candidate, the day-of-year mod still returns the same cluster every day until the pool grows — acceptable.
3. **What if the SSR cluster differs from the client cluster on the safety-net path?** Day-of-year is deterministic across server and client, but the client safety-net path uses session-seed. After a lens/category/PTR reset the cluster identity may change. This is the intended fallback behavior; the SSR-bake path is the contract, the client fetch is the safety net.
