# V11.18.21 — Surgical fix to `/api/discover/clusters` (timeout / silent-fail)

**Tag:** V11.18.21
**Predecessor:** V11.18.20 (SSR-bake cluster card — reverted on observation
that the underlying endpoint was timing out for months)
**Date:** 2026-06-10
**Scope:** Single-file change in `src/pages/api/discover/clusters.ts`.
Four surgical edits that take the endpoint from "hangs ≥ 40s and is
masked by edge SWR" to "responds in 3–5s end-to-end, cacheable for an
hour at the edge."
**Status:** Files saved; pending operator commit + tag. NUFORC ingest,
classifier-daily, patterns-counts cron all unchanged. No DB migrations.
No client-side or UI changes.

---

## TL;DR for the operator

The Geographic Cluster card has been failing to appear in `/discover`
for months. Root cause (full trace in
`docs/CLUSTERS_ENDPOINT_DIAGNOSIS.md`): the endpoint built a Haiku
finding for every `(state, category)` group with ≥ 3 reports in the
last 7 days — 64 geographic clusters + 1 temporal burst = **65 serial
Haiku calls** per cold request, each with a 12-second client-side
timeout — and then sliced to top 5 at the end, throwing away 92% of
the work. The handler's wall-clock was 98 s best-case, well past
Vercel's ~15 s function ceiling, so the function got killed mid-loop
on every cold execution; the edge stale-while-revalidate window
masked the bug for ~60 s after the rare cold success.

The DB queries underneath are healthy (~1.6 s total). The fix is
purely the serialization pattern + over-generation.

Four changes in one file:
1. Sort + slice candidates to top 8 **before** the Haiku loop.
2. `Promise.all` the per-cluster Haiku call + rep-report fetch.
3. Bump CDN cache 60 s → 1 hour with 5-minute SWR.
4. `export const config = { maxDuration: 30 }` for belt+suspenders.

Expected new response time: **3–5 s cold, <100 ms warm (edge cache).**

API response shape unchanged. The sessionSeed % 5 client picker in
`discover.tsx` continues to work — it now picks from the top 5 of
the top 8 highest-volume clusters, which is what the variety logic
was always designed around.

## Per-change diff summary

### Change 1 — Sort + slice BEFORE the Haiku loop (geographic arm)

**File:** `src/pages/api/discover/clusters.ts` (line ~439 area).

**Before (pseudocode):**

```ts
for (var keyG of Object.keys(geoGroups)) {
  var group = geoGroups[keyG]
  if (group.length < 3) continue
  // ...
  var haikuBody = await generateClusterFinding({...})  // 12s timeout
  var repReports = await loadClusterReports(...)       // ~80ms
  clusters.push({...})
}
// ...later...
clusters = clusters.slice(0, 5)
```

64+ iterations, all serial, only 5 of them survive. Inner-loop work
on items 6..65 is pure waste.

**After (V11.18.21):**

```ts
var topGeographic = Object.keys(geoGroups)
  .map(function (k) { return { key: k, group: geoGroups[k] } })
  .filter(function (e) { return e.group.length >= 3 })
  .sort(function (a, b) { return b.group.length - a.group.length })
  .slice(0, 8)   // headroom on the 5-card pool

var geographicCards = await Promise.all(topGeographic.map(async function (entry) {
  // ...one Haiku + one rep-report fetch per cluster, in parallel...
}))
```

**Why it matters.** Drops the geographic Haiku count from 64 → 8 (an
87.5% reduction). The 5-card pool is fed from the top 8 highest-
volume clusters — exactly the right shape for the
`sessionSeed % length` variety picker downstream in `discover.tsx`.

### Change 2 — Promise.all the per-cluster work

Per-cluster, the `generateClusterFinding(...)` Haiku call and the
`loadClusterReports(...)` Supabase round-trip are independent of each
other (one talks to Anthropic, the other to Supabase). Across
clusters, every cluster's pair is independent of every other.

**Before (per-cluster serial):**

```ts
const finding = await generateClusterFinding(grp)
const reps = await loadClusterReports(grp, ...)
clusters.push({ ...grp, finding, reps })
```

**After (per-cluster parallel, plus outer Promise.all across the
8-cluster pool):**

```ts
const enriched = await Promise.all(
  topGeographic.map(async (entry) => {
    const [finding, reps] = await Promise.all([
      generateClusterFinding(entry.group).catch(() => null),
      loadClusterReports(...),
    ])
    return { ...entry, finding, reps }
  })
)
```

Same shape applied to the temporal-burst arm (sliced to top 4
candidates — that arm is already small but the pattern is identical
for consistency and future-proofing).

**Why it matters.** Total Haiku wall-clock collapses from
`8 × max(Haiku) ≈ 24 s` to `1 × max(Haiku) ≈ 3 s`. Same for the
Supabase fetches. With both arms parallelised the cold-path budget is
~3–5 s end-to-end, comfortably inside Vercel's function ceiling.

### Change 3 — CDN cache header

**Before:**
```ts
res.setHeader('Cache-Control', 's-maxage=60, stale-while-revalidate=300')
```

**After (V11.18.21):**
```ts
res.setHeader('Cache-Control', 'public, s-maxage=3600, stale-while-revalidate=300')
```

**Why it matters.** Geographic clusters change on weekly-rollup
cadence; (state, category) shape drifts over hours/days, not seconds.
A 1-hour edge cache slashes load on the function dramatically while
still surfacing brand-new state clusters within an hour of the
shape shifting. `public` so Vercel's edge actually honours it
(default `private` defeats `s-maxage`). The original 60 s TTL was
tuned in V11.18.13 to a tight QA loop on the
`representative_reports` shape change; that constraint is long gone.

### Change 4 — `maxDuration` config

**Added at file bottom:**
```ts
export const config = {
  maxDuration: 30,
}
```

**Why it matters.** Default Vercel Pro Node serverless ceiling is
10–15 s. With Changes 1+2 the endpoint should respond in 3–5 s, so
30 s is pure headroom — if a single Haiku call stalls and hits its
own 12 s `AbortController` timeout, the function still has 18 s of
room before the platform kills it. No other route on
`/api/discover/*` carries `maxDuration`; this is local to clusters
and trivially reversible.

## Spot-check sequence

After deploy lands (~2–3 min for the Vercel build):

### 1. The 30-second curl test — response time

```bash
curl -sS -o /tmp/c1.json \
  -w 'HTTP %{http_code} bytes=%{size_download} ttfb=%{time_starttransfer}s total=%{time_total}s\n' \
  --max-time 30 \
  "https://www.discoverparadocs.com/api/discover/clusters?cb=$(date +%s)"
```

The `?cb=$(date +%s)` cache-buster guarantees a cold execution. Expected:

```
HTTP 200 bytes=<a few kB> ttfb=<3-5>s total=<3-5>s
```

If `total` is ≥ 25 s or the response is empty, something is wrong —
fall back to V11.18.20 by reverting the four edits and re-deploying.

### 2. Verify the response shape is unchanged

```bash
jq 'keys, (.clusters | length), (.clusters[0] | keys)' /tmp/c1.json
```

Expected: top-level `["clusters"]`, length 0–5, and the per-cluster
object has the same field set as V11.18.20 (`id`, `type`,
`cluster_type`, `type_label`, `headline`, `body`, `baseline_text`,
`category`, `category_label`, `report_count`, `time_range`,
`location_summary`, `linked_report_ids`, `representative_reports`,
`generated_at`, `headline_legacy`, `subheadline_legacy`).

### 3. Confirm the warm path is sub-100ms

Immediately re-run the curl WITHOUT the cache-buster:

```bash
curl -sS -o /tmp/c2.json \
  -w 'HTTP %{http_code} bytes=%{size_download} ttfb=%{time_starttransfer}s total=%{time_total}s\n' \
  --max-time 5 \
  "https://www.discoverparadocs.com/api/discover/clusters"
```

Expected `total` under 200 ms — that's Vercel's edge serving the
1-hour-cached response. If it's ≥ 1 s, the `Cache-Control` header
isn't being respected (check that `public` made it through any
Vercel `headers` config layer).

### 4. Visual spot-check in production

Open `https://www.discoverparadocs.com/discover` in an incognito
window. Scroll the feed. The Geographic Cluster card should appear
within the first ~30 cards. Refresh the page (same session) — the
SAME cluster card should appear (sessionSeed is stable per session).
Open a NEW incognito window — a DIFFERENT cluster card may appear
(different sessionSeed). This confirms the variety picker still
works on the 5-card pool.

### 5. Vercel function logs spot-check

```bash
vercel logs https://www.discoverparadocs.com/api/discover/clusters --since=10m
```

Expected pattern per cold invocation:
- 0 `[Clusters] Error:` lines.
- 8–12 `[loadClusterReports]` lines (8 geographic + up to 4 burst),
  with `total: 3` on most and structured `cluster=geo-…` keys.
- No `FUNCTION_INVOCATION_TIMEOUT` errors.

If you see `FUNCTION_INVOCATION_TIMEOUT` even once, raise
`maxDuration` to 45 and re-deploy. (Should not happen — 30 s is
~10× the realistic cold-path budget.)

## Uncertainty

- **Haiku call latency.** Assumed P50 of 1–3 s for the small prompt
  in `cluster-finding.service.ts`. Not directly measured for this
  specific path on the post-V11.18.21 code. If Haiku P50 has drifted
  to 5+ s, the cold-path budget swells to ~7 s — still well inside
  the 30 s `maxDuration` ceiling but worth noting if curl shows >7s
  consistently.
- **Vercel function ceiling.** Diagnosis assumed Pro Node default of
  10–15 s. `vercel.json` only configures `admin/` and `cron/` paths,
  so the default applies. `maxDuration: 30` overrides this
  explicitly, so the assumption is now moot for clusters
  specifically.
- **Edge-cache header.** Vercel respects `s-maxage` on Pro plans; if
  the team has a global `headers` override in `next.config.js` or
  `vercel.json` that strips `Cache-Control`, the 1-hour TTL won't
  stick. Spot-check #3 above catches this. No such override observed
  on a grep of the repo, but worth verifying.
- **NUFORC ingest cadence.** With NUFORC mass-ingest just completed
  (V11.17.x), the geographic cluster count of 64 is a peak-corpus
  shape. As steady-state drips set in, the pre-slice count may drop
  to 20–30 — the fix is still correct, just less dramatic in
  percentage terms.
