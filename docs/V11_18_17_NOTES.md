# V11.18.17 — Desktop /discover 3-column layout fix

**Tag:** V11.18.17
**Predecessor:** V11.18.16 (eyebrow re-rename, ClusteringCard substance backstop, LabPromo chip data path)
**Date:** June 2026
**Scope:** Fill the previously-empty left margin on desktop /discover
with a "On this week" rail of the 6 most-recent approved reports. No
DB migrations. No cron changes. No mobile / tablet visual changes.
**Status:** Code complete; pending operator commit + spot-check.

---

## 1. Bug

On desktop (≥1024px) `/discover` was rendering a constrained ~512–768px
centre column for the TikTok-grammar feed card, flanked by:

- An empty left margin (no content — viewport wasted)
- The Connected Cases right sidebar (V10.7.E.23) which often showed the
  "No related cases loaded yet" placeholder because the related-cards
  fetch runs lazily off the active card and the founder was hitting the
  page on a wide monitor before the fetch completed
- Massive vertical dead space above and below the card

The `sm:max-w-lg sm:mx-auto` constraint added in V11.18.13 preserves the
mobile portrait grammar at tablet sizes — that part of the decision
stays. The miss was: nothing actually filled the lg+ viewport.

Founder feedback: *"this looks awful on anything other than mobile."*

## 2. Fix — 3-column desktop layout

### 2.1 Structure

The lg+ row in `src/pages/discover.tsx` is now a 3-column flex:

```
┌────────────────┬──────────────────┬───────────────────┐
│  OnThisWeek    │  Feed card pane  │  Connected cases  │
│  (left rail)   │  (centre)        │  (right rail)     │
│                │                  │                   │
│  280px wide    │  lg:max-w-2xl    │  340px wide       │
│  hidden lg:    │  xl:max-w-3xl    │  hidden lg:       │
│  flex          │  unchanged       │  flex (existing)  │
└────────────────┴──────────────────┴───────────────────┘
```

- **Left rail (`OnThisWeekRail`, new):** 6 most-recent approved
  reports, sourced from `/api/homepage/recent-reports?limit=6` (the
  existing endpoint that powers the marketing-homepage ticker —
  s-maxage=30, 5min SWR). Each card: category eyebrow + location +
  relative timestamp + 2-line headline, with a 3px category-colour
  accent on the left border. Footer link to `/explore`.

- **Centre:** Unchanged. Existing feed card, snap behaviour,
  TikTok-grammar swipe, special-card injection, SSR bake — all
  untouched.

- **Right rail (`Connected cases`, pre-existing V10.7.E.23):** Still
  populated by the active-card-driven `/api/discover/related-cards`
  fetch. The "No related cases loaded yet" placeholder is the
  pre-fetch state for the very first card; once the user has dwelled
  on a card for a tick, the fetch lands and the rail populates. No
  change in V11.18.17 — confirmed working, just visually less lonely
  now that there's a left-side counterpart.

### 2.2 Files

- **New:** `src/components/discover/OnThisWeekRail.tsx` — `hidden
  lg:flex` aside, `w-[280px] xl:w-[300px]`. Internally scrollable.
  Skeleton shimmer on first paint. Empty-state message if the API
  returns zero rows. Footer link to `/explore` so the rail is anchored.

- **Modified:** `src/pages/discover.tsx`
  - Import added next to the SkeletonCard import.
  - `<OnThisWeekRail />` mounted as the first child of the existing
    `<div className="flex-1 flex lg:max-h-[calc(100vh-6.5rem)]">` row.
  - Loading state (centred SkeletonCard branch above the main return)
    now also renders the rail so the page doesn't snap from "narrow
    centred skeleton" to "3-column populated layout" once the feed
    lands. Centre column in the loading branch picks up the new
    `lg:max-w-2xl xl:max-w-3xl` widths to match the loaded state.

### 2.3 Mobile / tablet — explicitly unchanged

- Mobile (<640px): full-bleed cards, TikTok-grammar swipe — exactly
  as before. Both rails are `hidden lg:flex`.
- Tablet (640–1023px): centre column constrained by `sm:max-w-lg
  sm:mx-auto` (V11.18.13) — exactly as before. Rails are still hidden.
- Desktop (≥1024px): 3-column layout kicks in. Both rails appear.
  Centre column keeps its `lg:max-w-2xl xl:max-w-3xl` cap so the
  TikTok-grammar card doesn't balloon — the rails fill the rest.

### 2.4 Data path — left rail

```
OnThisWeekRail (mount useEffect)
  → fetch('/api/homepage/recent-reports?limit=6')
  → setReports(data.reports)
  → render 6 cards linking to /report/{slug}
```

Endpoint is public, no auth, edge-cached (s-maxage=30, SWR=300), so the
fetch never blocks first paint. Failure mode: the rail shows the empty-
state message; the rest of the page is unaffected.

## 3. V11.18.16 invisibility diagnostic

Founder reports the V11.18.16 cluster-substance backstop and LabPromo
chip fixes aren't visible. Three hypotheses, in descending likelihood:

1. **V11.18.16 hasn't shipped.** Confirm by running:

   ```
   git log --oneline -n 5
   ```

   If V11.18.16 isn't tagged or hasn't been pushed, the Vercel deploy
   won't have picked it up. This is the most plausible cause —
   V11.18.16 marked "Code complete; pending operator commit" in the
   header of `docs/V11_18_16_NOTES.md`, and there's no indication the
   commit landed.

2. **Vercel deploy hadn't propagated.** Check the deploys list:

   ```
   vercel ls discoverparadocs
   ```

   Look for the V11.18.16 commit SHA in the most-recent deploy. If
   it's a different SHA or the deploy is "Building", the founder is
   seeing the previous release.

3. **CDN edge cache.** `/api/discover/clusters` dropped from
   s-maxage=600 → 60 in V11.18.13. Sixty seconds is fast, but if the
   founder hard-refreshed within that window, the edge could still
   have been serving a stale response. Verify with:

   ```
   curl -s 'https://www.discoverparadocs.com/api/discover/clusters' \
     | jq '.clusters[0] | {id, cluster_type, rep_count: (.representative_reports | length)}'
   ```

   `rep_count` should be ≥3 post-V11.18.16. If it's 0, the backstop
   isn't running — meaning V11.18.16 isn't deployed (re-check (1)).

4. **Console check.** Hard-refresh, open DevTools, scroll to a
   ClusteringCard. If it's still empty, the
   `console.warn('[ClusteringCard] empty representative_reports...')`
   diagnostic added in V11.18.16 will fire. If it doesn't fire AND the
   card is empty, the V11.18.16 JS bundle isn't in the page — which
   loops back to (1) or (2).

## 5. Uncertainty / open notes

- **Right rail still says "No related cases loaded yet" on first
  paint.** This is by design (the related-cards fetch is anchored to
  the active card and runs after first paint), and V11.18.17 doesn't
  change it. If the founder finds the empty-state copy still feels
  unfinished even with the OnThisWeekRail on the left side, a follow-
  up could pre-fetch the first card's connections in
  `getServerSideProps` and bake them into props alongside the
  Finding + OnThisDate. Out of scope for this release.

- **Rail content overlap with feed.** The OnThisWeekRail surfaces
  recent reports independently of personalisation, so a card that
  appears in the user's feed-v2 ranker could also appear in the rail.
  Acceptable — the rail is a "what's landing in the archive" surface
  rather than a personalised feed. If overlap becomes annoying we can
  filter the rail against the user's seen-ids set, but Helena rule:
  ship the smaller intervention first.

- **No A/B test framing.** This is a desktop-only layout fix; the
  founder explicitly framed it as "looks awful on anything other than
  mobile." Treating as a quality fix, not a conversion experiment.
