# V10.9 — Explore-map aggregation at mass-ingest scale (Design)

**Status:** Design draft. Pending Chase sign-off + implementation post-V10.8.
**Authors:** SME panel (UI/UX, Brand, Data Viz Engineer, Cartographer) convened during V10.8.I.
**Goal:** Make `/explore?mode=map` work cleanly when the corpus grows from 107 reports to 1M+ during mass ingestion, while honoring the fundamental difference between *precise* and *synthetic* location data.

---

## The problem in one sentence

When the corpus is full of country-only reports (centroid-fallback coords), pin-style rendering piles them all at the country centroid — looks like a bug, undermines credibility, hides the actual geographic story.

## Concrete symptom (V10.8.I screenshot)

After backfilling 97 existing reports, ~57 of them have `coords_synthetic=true` because the source provided only country (or country + state). On the explore map, dozens of these stack at the US country centroid (Kansas-ish), producing a "56" cluster bubble that's actually 56 *different US-country-only reports*, not 56 reports happening near Kansas. The cluster is real but the geography is fake.

This will get worse: every NDE/OBE source that has "United States" but no city contributes another point to the Kansas pile. At mass-ingest scale we could see 50K+ reports stacked on the US centroid alone.

---

## Principles (SME panel consensus, V10.8.I)

1. **Honesty over precision-by-default.** Synthetic centroid coords NEVER render as pins. They render as region highlights, fuzzy circles, or aggregation buckets.
2. **Precision tier drives visual treatment.** One ruleset applied everywhere (report page AND explore map).
3. **Two-layer aggregate architecture.** Precise pins ON TOP of region choropleth — both visible together.
4. **Server-side aggregation for mass-ingest scale.** Never ship raw country-precision rows to the client; aggregate to country/state count buckets.
5. **`coords_synthetic` is the canonical signal.** V10.8.C already shipped this column — V10.9 just uses it.

---

## The architecture

### Layer 1 — Precise-pin layer (existing)

Source: `reports` where `coords_synthetic=false`. These have real GPS/city geocodes.

Rendering: existing Supercluster + circle-marker layers in `MapContainer.tsx`. Already works. No change.

At mass-ingest scale this layer might have ~30% of total reports (the BFRO/NUFORC/MapTiler-geocoded ones). Even at 300K precise pins, Supercluster handles that fine.

### Layer 2 — Region choropleth layer (NEW)

Source: aggregated counts per `country_code` and `(country_code, state_province)` for rows where `coords_synthetic=true`. Computed server-side.

Two sub-layers, zoom-aware:

**2a. Country choropleth** (visible at zoom 0-4)
- Country polygons (GeoJSON from Natural Earth admin0, ~100KB)
- Fill color intensity = log(country_count)
- Hover/click tooltip shows count + top categories
- Brand palette: dark base, purple→cyan gradient by count

**2b. State choropleth** (visible at zoom 4-7, US/CA/UK/AU only initially)
- State polygons (GeoJSON admin1 subset, ~200KB)
- Same color encoding
- Long-tail countries fall back to country choropleth

**Toggle controls** (top-right corner)
- "Pin locations only" (precise layer alone, default for searches with city queries)
- "Region density" (choropleth alone, useful for "what countries have most reports")
- "Combined" (both, default for global view)

### Layer 3 — Heatmap (existing, unchanged)

Layer-2 doesn't replace heatmap. They coexist: heatmap shows raw density (any precision), choropleth shows clean per-region counts.

## Schema + API

### Aggregate query (cached materialized view)

```sql
CREATE MATERIALIZED VIEW report_region_counts AS
SELECT
  country_code,
  state_province,
  category,
  count(*) AS report_count
FROM reports
WHERE
  status = 'approved'
  AND coords_synthetic = TRUE
  AND country_code IS NOT NULL
GROUP BY country_code, state_province, category;

CREATE INDEX ON report_region_counts (country_code);
CREATE INDEX ON report_region_counts (country_code, state_province) WHERE state_province IS NOT NULL;
```

Refresh nightly via cron. At mass-ingest scale the view is < 10K rows even with 1M reports (210 countries × ~50 states × 11 categories).

### API endpoint

```
GET /api/map/region-counts?level=country
GET /api/map/region-counts?level=state&country=US
```

Response shape:
```json
{
  "level": "country",
  "buckets": [
    { "code": "US", "name": "United States", "total": 12453, "by_category": { "ufos_aliens": 4321, "ghosts_hauntings": 2891, ... } },
    { "code": "GB", "name": "United Kingdom", "total": 1823, "by_category": {...} },
    ...
  ]
}
```

### GeoJSON assets

- `public/geo/admin0-countries.geojson` — Natural Earth admin0 simplified (~100KB)
- `public/geo/admin1-us-ca-gb-au.geojson` — US states + CA provinces + UK home nations + AU states (~200KB)
- Both static, loaded once and cached aggressively

Fetched on map mount, joined with the API counts client-side to produce the colored polygons.

## Recommended ship order

1. **Aggregate query + materialized view** — pure SQL. Test with current 107-row corpus, validate counts.
2. **`/api/map/region-counts` endpoint** — wraps the view, simple GROUP BY query.
3. **GeoJSON assets** — bundle Natural Earth simplified at `public/geo/`. CC-BY license, public domain compatible.
4. **MapContainer choropleth layer** — new MapLibre `fill` layer reading the GeoJSON joined with API counts. Zoom-conditional visibility.
5. **Toggle UI** — three-button control (Pins / Regions / Combined) in the map toolbar.
6. **Removal of synthetic pins from precise layer** — once choropleth is live, filter `coords_synthetic=false` on the precise-pin source so synthetic points stop pile-up entirely.

Total estimated work: ~1.5 sessions.

---

## Open questions

1. **Continuous refresh vs nightly batch?** Mass-ingest is bursty. If a 10K-row batch finishes at 2am, the dashboard's region counts won't reflect it until next refresh. Acceptable. If we want fresher: REFRESH MATERIALIZED VIEW after every ingestion run (still cheap at 1M rows).
2. **Category color encoding on choropleth?** Single-category color is clean. Multi-category color (pie chart per region) is informative but cluttered. Recommend single-category (most-common in that region) with hover-card showing the breakdown.
3. **Mobile UX for choropleth?** Country polygons are fat tap targets on mobile — good. State polygons less so. Recommend: state choropleth visible only at zoom 5+ on mobile.
4. **Should the precise-pin layer also be filtered by category when a category filter is active?** Yes — already done. Just extend to the choropleth (filter the API by category).
5. **What about reports with `country_code` but no state where we DO have city-level coords from MapTiler?** Those are `coords_synthetic=false` (precise pins). They flow through Layer 1. No change.
