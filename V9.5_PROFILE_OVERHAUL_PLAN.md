# V9.5 — Profile / Account Overhaul

**Status:** Ready to execute
**Source:** Expert panel review + Chase's bug list
**Last updated:** 2026-05-07

---

## The bugs Chase reported

1. **Account Settings + Subscription nav broken** — clicking the row changes the URL to `/dashboard/settings` or `/dashboard/subscription`, but the page content stays as Profile.
2. **Public Profile shows "this researcher's profile is private"** — even for the signed-in user viewing their own page.
3. **NotificationToggle on desktop says "Disabled in iOS Settings"** — that copy only makes sense on iOS PWA.
4. **Stats box has Hypotheses + Corroborations** — neither is tracked in our current model. Needs replacement.

## Root causes (audit verdict)

1. **Nav bug:** Two co-conspirators.
   - `src/pages/dashboard/settings.tsx` has a **duplicate type import** at lines 35–36 (`import type { PhenomenonCategory }` twice). This produces `TS2300: Duplicate identifier`. In production builds, the page fails to compile; in dev, Next.js logs but still serves — except a layout swap from default `Layout` → `DashboardLayout` against a broken Component causes the router to update the URL while the previous page stays mounted.
   - Both `/dashboard/settings` and `/dashboard/subscription` live under `CUSTOM_LAYOUT_PREFIXES = ['/dashboard']` in `_app.tsx`. That's fine, but the layout mismatch amplifies any error during navigation.
2. **Public Profile "private":** `/researcher/[username].tsx` checks a `profile_visibility` column / RLS policy. New users default to `private`. There is no UI anywhere to flip this. So every user sees their own public profile as "private."
3. **Desktop "iOS Settings" copy:** `NotificationToggle.tsx` hardcodes "Re-enable in iOS Settings → Paradocs" when `permission === 'denied'`. No platform branch.
4. **Stats:** `Hypotheses` + `Corroborations` are residue from an older social model that we never shipped. We currently track: saves, reports filed, streak days, and join date.

## Bonus issue surfaced during audit

- **URL shape** — `/dashboard/settings` and `/dashboard/subscription` are split between a Profile page and a "dashboard" namespace that we no longer use elsewhere. Best practice is to consolidate under `/account/*`.

## Plan — 4 phases

### Phase 1 — Critical fixes (ship first, unblocks Chase)

**P1.1** Drop the duplicate `import type { PhenomenonCategory }` in `src/pages/dashboard/settings.tsx`.

**P1.2** Make `NotificationToggle.tsx` platform-aware. Detect iOS standalone via `window.matchMedia('(display-mode: standalone)')` + `/iPad|iPhone|iPod/.test(ua)`. If iOS standalone → "Re-enable in iOS Settings → Paradocs." Else → "Re-enable in your browser's site settings." Add a chrome/firefox/safari-agnostic copy variant.

**P1.3** Add a `profile_visibility` toggle in Settings (`public` | `private`). Default for **existing** users stays whatever they had; **new** users default to `public` (researchers want to be discoverable). Show a one-line explainer on the public profile page when private.

**P1.4** Replace the Profile stats box. New stats (left to right):
- **Saved** (saved_count from `/api/user/stats`)
- **Reports** (report_count)
- **Streak** (current streak days from `paradocs_streak` localStorage + server bridge)
- **Joined** (year of `created_at`)

Drop `hypothesis_count` and `corroboration_count` from the response shape entirely.

**Estimated scope:** ~250 LOC across 4 files. One migration not needed (column exists; we just need a setter).

---

### Phase 2 — Visual unification

**P2.1** Single column layout on Settings + Subscription (currently inconsistent — Subscription has a 4-column tier grid that overflows on mobile).

**P2.2** Kicker headers across all account pages: small uppercase eyebrow + bold title, matching Profile's identity card.

**P2.3** Sticky bottom save bar on Settings (panel ask): instead of inline "Save Changes" buttons per section, a single sticky bar pinned to the bottom of the viewport that activates when the user has unsaved changes.

**P2.4** Subscription: replace tier grid with a hero card showing **current state** ("You're on Free" / "You're on Pro — renews $X on Y") and a single primary CTA. Move the comparison grid below the fold.

**Estimated scope:** ~400 LOC across 3 files.

---

### Phase 3 — IA + privacy (URL restructure)

**P3.1** Move `/dashboard/settings` → `/account/settings`, `/dashboard/subscription` → `/account/subscription`. Add 301 redirects from the old paths. Update `CUSTOM_LAYOUT_PREFIXES` in `_app.tsx` to include `/account`.

**P3.2** Update all internal `<Link>`s in `profile.tsx`, `DashboardLayout.tsx`, footer, and any other entry points.

**P3.3** Add a **Your Data** section in Settings: download account JSON, delete account (deferred to a confirmation flow we'll build later).

**P3.4** Add a **Visibility** section in Settings with the explainer: "Public profile means anyone with your @username can view your saves and rank. We never share your email."

**P3.5** Anchor-link nav at the top of Settings: `Profile · Notifications · Visibility · Data` jumping to in-page sections. Sticky on desktop.

**Estimated scope:** ~300 LOC across 6 files + 1 redirect config.

---

### Phase 4 — Subscription depth

**P4.1** Usage meters on Subscription: e.g., "12 of 50 saves this month" if there are tier limits.

**P4.2** **Manage billing** button → Stripe customer portal session.

**P4.3** Cancellation flow with retention: if a user clicks cancel, show the "What you'll lose" panel before we redirect to Stripe.

**P4.4** Receipts/invoices list (pulled from Stripe).

**Estimated scope:** ~350 LOC across 2 files + 1 new API route.
