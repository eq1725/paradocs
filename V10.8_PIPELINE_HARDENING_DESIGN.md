# V10.8 — Ingestion Pipeline Hardening (Design)

**Status:** Design draft — pending Chase sign-off before any code.
**Author:** Paradocs V10.7 session, May 13, 2026.
**Purpose:** Replace the ad-hoc per-adapter extraction logic with three unified, well-tested utilities (date, location, validation) so mass-ingest produces consistent, audit-trail-backed data without per-row QA.

---

## Context — what the current pipeline gets wrong

Audit of all 15 ingestion adapters (`src/lib/ingestion/adapters/*.ts`):

**Date extraction is fragmented and lossy.**

| Adapter | Today's behavior |
|---|---|
| OBERF | Structured "Date of Experience" → fallback regex captures ONLY year from prose. "On April 28th 2007" gets stored as `2007-01-01, precision='year'`. |
| NDERF | Same shape as OBERF — year-only prose fallback. |
| BFRO, GhostsOfAmerica, NUFORC | Per-adapter date parsers. Each maintained separately. |
| News | Stores publication date as `event_date_precision='exact'`. This is almost always wrong — pub date is when the article was published, not when the event happened. |
| Wikipedia | Hardcoded to `'year'` regardless of what the lede contains. |
| Reddit, Reddit-v2, IANDS, Erowid, Shadowlands, YouTube | Hardcoded to `'unknown'`. Never even attempt extraction. |

**Location handling is similarly fragmented.** No unified normalizer. Reports with `country='United States'` (no state/city/coords) account for ~52% of the current corpus and render as "Location unknown" without a render-time hack. At mass-ingest scale across 200+ countries this is unmanageable.

**No ingestion-time validation.** The pipeline silently accepts impossible dates, lat/lng outside valid ranges, country-state mismatches, and rows missing required fields.

## V10.8.A — Unified date extractor

**Location:** `src/lib/ingestion/utils/extract-date.ts` (new).

**Public surface:**

```ts
export type DatePrecision = 'exact' | 'month' | 'year' | 'unknown'

export interface ExtractedDate {
  /** ISO YYYY-MM-DD, always populated even at lower precisions (1st of month/year as sentinel). */
  date: string | null
  precision: DatePrecision
  /** Audit trail — how we arrived at this answer. */
  source: 'structured' | 'prose-monthname' | 'prose-numeric' | 'prose-year' | 'none'
  /** Original text that matched, for debugging the regex layer. */
  matchedText?: string
  /** True if approximate markers ('circa', 'around', 'early 2000s') were present. */
  approximate?: boolean
}

export interface ExtractDateOptions {
  /** Pre-cleaned structured field value (e.g. OBERF 'Date of Experience'). Tried first. */
  structured?: string | null
  /** Free-form prose to scan if structured is missing or unparseable. */
  prose?: string | null
  maxYear?: number  // default: current year + 1
  minYear?: number  // default: 1800
}

export function extractDate(opts: ExtractDateOptions): ExtractedDate
```

**Algorithm (cascading priority):**

1. **Structured field parsing.** Try `MM/DD/YYYY`, `YYYY-MM-DD`, `Month D, YYYY`, `Month YYYY`, `YYYY`. Handle sentinel zeros (`04/00/2007` → month precision). If structured value is present but unparseable, log + fall through (don't crash).

2. **Prose month-name extraction.** ★ NEW capability. Regex:

   ```
   (?:on\s+|in\s+)?           # optional "on" / "in" preamble
   (Jan|January|Feb|February|...|Sept?|...|December)
   \s+
   (\d{1,2})(?:st|nd|rd|th)?  # day with optional ordinal suffix
   (?:,)?\s+
   (\d{4})                    # 4-digit year
   ```

   Also UK form (`28 April 2007`) and month-only form (`April 2007`). Bias toward the FIRST match in prose — narrative-opening dates are almost always the event date.

3. **Prose numeric date.** `MM/DD/YYYY` or `YYYY-MM-DD` patterns embedded in prose, anchored by punctuation.

4. **Prose year-only.** `\b(18\d\d|19\d\d|20\d\d)\b` but require date-like context (preceded by "in"/"around"/"during"/"since" or followed by punctuation) — avoids hitting "1,200 reports".

5. **Approximate-marker detection.** "circa", "around", "about", "approximately", "estimated", "early/mid/late 2000s" → set `approximate=true`, downgrade precision (e.g. "early 2000s" → `2002-01-01, precision='year', approximate=true`).

6. **Validation gates.** Reject year > maxYear or < minYear. Reject impossible dates (Feb 30, etc.). No signal → `{date: null, precision: 'unknown', source: 'none'}`.

**Why this is the killer change:** Adding the prose-monthname regex captures the pit-bull case ("On April 28th 2007") cleanly with NO LLM call. The hardest data fix in the current pipeline reduces to ~150 lines of well-tested regex.

**Risks & mitigations:**

- **"April Fool's Day" false positive.** Require the month to appear within 30 chars of a 4-digit year. Add a tiny stopword list for "Fool's", "showers" (April), "fresh" (May), etc. — covered by fixture tests.
- **Multiple dates in prose.** Bias to the FIRST date. When in doubt, log both, take the first. Fixtures cover "the building was constructed in 1923 but the haunting started in 2018" → should extract 1923 if first, 2018 if first. Adapter can pass narrowed prose (e.g. first paragraph only) for stricter "event date" semantics.

## V10.8.B — Wire extractDate into every adapter + regression fixtures

**Files touched:** All 15 adapters in `src/lib/ingestion/adapters/`.

**Per-adapter migration:**

| Adapter | Today | After V10.8.B |
|---|---|---|
| OBERF | structured + year-only prose | `extractDate({ structured: getField(['Date of Experience']), prose: content })` |
| NDERF | structured + year-only prose | `extractDate({ structured: <field>, prose: <html-stripped lede> })` |
| BFRO | per-adapter parser | `extractDate({ structured: bfroDateField, prose: bfroDescription })` |
| GhostsOfAmerica | per-adapter parser | `extractDate({ structured: goaHeader, prose: storyText })` |
| NUFORC | per-adapter parser | `extractDate({ structured: occurredField, prose: descripField })` |
| News | `precision='exact' = pub_date` (BUG) | `extractDate({ prose: articleBody })`. Pub date moves to `source_published_at` field (separate migration). |
| Wikipedia | year-only | `extractDate({ prose: lede })`. Wikipedia ledes are date-rich. |
| Reddit + Reddit-v2 | `'unknown'` (no attempt) | `extractDate({ prose: postBody })`. Captures ~30-60% of dates. |
| IANDS, Erowid, Shadowlands, YouTube | `'unknown'` (no attempt) | `extractDate({ prose: description })`. |

**Adapter-specific structured field paths stay** — each source names its date field differently (e.g. OBERF's "Date of Experience" vs NUFORC's "Occurred"). Only the parsing logic is centralized.

**Schema migration alongside B:**

- Add `reports.source_published_at TIMESTAMPTZ NULL` — for news (and any pub-date-bearing source). Existing news rows backfill: copy current `event_date` → `source_published_at`, then re-extract `event_date` from prose in B's run.
- Add `reports.event_date_extracted_from TEXT NULL` — audit field storing the `source` enum from `ExtractedDate.source`. Lets the admin panel filter ingests by extraction confidence.

**Regression test:** `test/ingestion/extract-date.test.ts`.

Table-driven fixture set, ~40 entries drawn from real source samples:


## V10.8.C — Unified location normalizer

**Files:**

- `src/lib/ingestion/utils/normalize-location.ts` (new)
- `src/lib/ingestion/utils/country-centroids.json` (new, ~250 countries, ~6KB)
- `src/lib/ingestion/utils/state-centroids.json` (new, US states + Canada provinces + UK home nations + AU states, ~80 entries)
- Migration `20260514_geocode_cache.sql` — `geocode_cache(query TEXT PK, lat NUMERIC, lng NUMERIC, accuracy TEXT, geocoded_at TIMESTAMPTZ)`
- Hook in `src/lib/ingestion/engine.ts` immediately before insert.

**Public surface:**

```ts
export type LocationPrecision = 'exact' | 'city' | 'region' | 'country' | 'unknown'

export interface NormalizedLocation {
  city: string | null
  state_province: string | null
  country: string | null         // canonical name, e.g. "United States"
  country_code: string | null    // ISO 3166-1 alpha-2, e.g. "US"
  location_name: string | null
  latitude: number | null
  longitude: number | null
  location_precision: LocationPrecision
  /** True when coords came from a centroid fallback rather than precise geocoding. */
  coords_synthetic: boolean
}

export async function normalizeLocation(
  raw: Partial<NormalizedLocation>,
  options?: { geocoder?: 'maptiler' | 'none'; cache?: GeocodeCache }
): Promise<NormalizedLocation>
```

**Algorithm:**

1. **Country alias normalization.** Fold variants via the centroid JSON's alias list:
   - "USA", "U.S.A.", "U.S.", "America" → "United States" (US)
   - "UK", "U.K.", "Britain", "Great Britain" → "United Kingdom" (GB)
   - ~50 common aliases plus everything Natural Earth identifies

2. **State/province validation.** Verify the state belongs to the country (US states + CA provinces + UK home nations + AU states first; long tail in V2). State+wrong-country triggers `LOC_STATE_COUNTRY_MISMATCH` warning in D.

3. **Geocoding ladder:**

   | Inputs available | Action | Output precision | coords_synthetic |
   |---|---|---|---|
   | Valid lat/lng provided | Validate range, keep | based on other fields | false |
   | city + state(+country) | Geocode via MapTiler, cache by `city\|state\|country` key | `city` | false |
   | state + country | State centroid from JSON | `region` | true |
   | country only | Country centroid from JSON | `country` | true |
   | nothing usable | Skip | `unknown` | n/a |

4. **Validation gates** (also enforced again in D):
   - Reject coords outside `[-90, 90] × [-180, 180]`.
   - Reject `(0, 0)` if no country (almost always a parsing bug).

5. **`coords_synthetic` field.** New column. Renders use this to softer-style markers when the coord is approximated. The current render-side `COUNTRY_CENTROIDS` fallback in `ReportPageV2` can be REMOVED once C ships — every row will have coords from the DB, so the page is pure pass-through.

**Country-centroids JSON shape:**

```json
{
  "US": {
    "name": "United States",
    "lat": 39.8283, "lng": -98.5795,
    "aliases": ["USA", "U.S.A.", "America"]
  },
  "UA": {
    "name": "Ukraine",
    "lat": 48.3794, "lng": 31.1656,
    "aliases": ["UKR"]
  },
  ...
}
```

Source dataset: Natural Earth admin0 centroids (public domain). All ~250 countries; centroids hand-tweaked for ~30 most-common entries (e.g. UK uses England-center, not Irish Sea).

**Geocode cache table:**

```sql
CREATE TABLE geocode_cache (
  query TEXT PRIMARY KEY,                  -- normalized 'city|state|country' key
  lat NUMERIC NOT NULL,
  lng NUMERIC NOT NULL,
  accuracy TEXT,                           -- MapTiler accuracy label
  geocoded_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Hit-rate estimate: ~70% cache hit during mass-ingest of 1M reports (cities repeat heavily). ~300K MapTiler calls. Free tier is 100K/month; need a $50/month plan for the initial mass-ingest. Acceptable, flagged.

## V10.8.D — Ingestion validation gates

**Files:**

- `src/lib/ingestion/utils/validate-report.ts` (new)
- Migration `20260514_ingestion_audit.sql` — `ingestion_audit(id, report_id NULLABLE, adapter, severity, code, message, field, payload JSONB, created_at)`. Nullable report_id because errors block insert.
- New table `reports_quarantine` (or status enum extension `'quarantine'`) for hard-fail rows.
- Admin page `/admin/ingest-audit` shows warning/error rates by adapter + code.

**Public surface:**

```ts
export interface ValidationResult {
  ok: boolean
  warnings: ValidationFlag[]
  errors: ValidationFlag[]
}

export interface ValidationFlag {
  code: string
  message: string
  field: string
  severity: 'warning' | 'error'
}

export function validateReportBeforeInsert(report: ReportShape): ValidationResult
```

**Warning gates (soft-fail, log + insert):**

| Code | Trigger |
|---|---|
| `DATE_SENTINEL_EXACT` | `event_date_precision='exact'` but `event_date` matches `-01-01`. Indicates precision was over-claimed. |
| `DATE_FUTURE` | `event_date` > today. |
| `DATE_TOO_OLD` | `event_date` year < 1800 unless category permits (folklore/mythology). |
| `LOC_COUNTRY_NO_COORDS` | Country set but lat/lng null after normalizeLocation ran. Should be impossible after C ships — if it fires, normalize has a bug. |
| `LOC_COORDS_NO_COUNTRY` | Lat/lng present but country null. Reverse-geocode or flag. |
| `LOC_COORDS_ORIGIN` | `(0, 0)` — parsing bug indicator. |
| `LOC_STATE_COUNTRY_MISMATCH` | State doesn't belong to country (e.g. Texas + Mexico). |
| `TEXT_TITLE_GENERIC` | Title equals `original_title` or matches `^(Other|Untitled|Unknown)`. AI title generator didn't fire. |
| `TEXT_NARRATIVE_EMPTY` | `paradocs_narrative` null and source description >100 words. Analysis pipeline didn't fire. |
| `WITNESS_PROFILE_MISSING` | `witness_profile` null. Profile service didn't fire. |
| `CATEGORY_UNKNOWN` | `category` null or 'other'. Classifier didn't produce. |

**Error gates (block insert, quarantine):**

| Code | Trigger |
|---|---|
| `MISSING_REQUIRED` | No `source_url`, no `source_label`, or empty `description`. |
| `DUPLICATE_ORIGINAL_ID` | `original_report_id` already exists with different content (dedup race). |
| `DATE_INVALID` | `event_date` doesn't parse as `YYYY-MM-DD`. |
| `LOC_OUT_OF_RANGE` | lat/lng outside valid range. |

**Audit log shape per validation result:**

```sql
INSERT INTO ingestion_audit (
  report_id,          -- null if blocked (quarantined)
  adapter,            -- 'oberf' | 'nderf' | ...
  severity,           -- 'warning' | 'error'
  code,               -- 'DATE_SENTINEL_EXACT' | ...
  message,            -- human-readable
  field,              -- 'event_date' | 'latitude' | ...
  payload             -- JSON dump of the offending values for forensics
);
```

**Admin observability:**

- `/admin/ingest-audit` — filter by adapter + code + date range. "OBERF emitted DATE_SENTINEL_EXACT on 47 of last 1000 rows" tells you immediately a regex regressed.
- `/admin/quarantine` — list of quarantined rows with the failing code. Quick triage workflow: review → either fix data inline (then re-validate) or hard-reject (mark `status='rejected'`).
- Dashboard tile: top-5 warning codes by frequency over the last 7 days. Mass-ingest is "healthy" when this is dominated by `LOC_COORDS_NO_COUNTRY` (rare edge case) and "concerning" when it's `DATE_SENTINEL_EXACT` (means an adapter is over-claiming).

## Recommended ship order

Different from the alphabet — A through D as a unified plan, but landed in dependency order:

1. **V10.8.A** — `extractDate` utility + unit tests. Pure new code, no existing-system risk. ~1 session.
2. **V10.8.B** — Adapter migration + fixture regression suite. Mechanically repetitive across 15 files but each change is isolated. ~1 session. Also includes the `source_published_at` migration for news.
3. **V10.8.D** — Validation gates wired into `engine.ts`. Cheap once A+B are in (warnings get easier to detect). ~0.5 session. Includes `ingestion_audit` table migration and admin page.
4. **V10.8.C** — Location normalizer + centroid JSONs + `geocode_cache` migration + MapTiler integration. Heaviest piece because of the new external dependency (MapTiler), the new persistent cache, and the `coords_synthetic` column on `reports`. ~1 session.
5. **V10.8.E** (optional) — Haiku-assisted date extraction for residual hard cases (precision='year' but month names visible). Ship only if A-D leave a measurable error rate. ~0.5 session.

**Total: 3.5-4 sessions for A-D, plus optional E.** Recommend landing each as a separate commit chain so each layer can be reviewed and tested before the next stacks.

## Test surface

**Unit tests (CI-gated):**

- `extract-date.test.ts` — 40+ table-driven fixtures from real source samples (one fixture per adapter type, plus adversarial cases). Asserts `{date, precision, source}`.
- `normalize-location.test.ts` — 20+ fixtures covering every continent + edge cases (country alias folding, state-country mismatch, origin-coord rejection).
- `validate-report.test.ts` — full warning/error matrix.

**Integration test:**

- Re-ingest a frozen fixture corpus (5 real reports per adapter type from a saved HTML snapshot) and assert every row passes validation with no warnings of `severity > info`. Snapshot the final DB row shape; a snapshot diff fails CI on accidental schema/extraction regressions.

**Manual QA gates before declaring V10.8 done:**

- Re-ingest the pit-bull report. Expected: `event_date='2007-04-28', event_date_precision='exact', event_date_extracted_from='prose-monthname', country='United States', country_code='US', latitude≈39.83, longitude≈-98.58, coords_synthetic=true, location_precision='country'`.
- Re-ingest one of each: OBERF, NDERF, BFRO, NUFORC, GhostsOfAmerica, Wikipedia, Reddit, IANDS, Erowid, Shadowlands, YouTube, News. Assert each produces `precision >= 'year'` (not 'unknown') wherever the source actually has date language.
- Render the pit-bull report. WHEN line should read "April 28, 2007" — the core acceptance criterion.

## Open questions for sign-off

1. **MapTiler $50/month for geocoding** — OK to expense once mass-ingest starts? Cheaper alternatives (Nominatim via OSM) exist but rate-limit aggressively and have less reliable accuracy. Recommendation: pay MapTiler, scope creep elsewhere.

2. **Quarantine vs hard-reject** — when an error gate fires (e.g. `MISSING_REQUIRED`), do we want a quarantine table or just `status='rejected'`? Recommend `status='quarantine'` enum value — same table, no new column write, distinct from `'rejected'` which implies admin already reviewed.

3. **News pub-date migration** — moving pub date from `event_date` to `source_published_at` affects ~15 news rows in the current corpus. Backfill these in V10.8.B (one-time SQL), then never store pub date in event_date again. Alternative: leave news adapter's behavior as-is for now (it's tiny corpus volume) and fix in a later pass. Recommend backfilling in B since the migration is trivial.

4. **Re-extraction of the 31 current bad rows** — your stated preference is "ready the pipeline, don't QA the current rows." Confirming we do NOT plan to re-ingest the 31 affected rows; they live with their current precision until they get naturally re-ingested at mass-ingest time. If you change your mind, the Haiku-assisted approach from V10.7.J (~$0.15 total) is the quickest path.

5. **Render-side `COUNTRY_CENTROIDS` table** — should be removed once V10.8.C ships and the DB always has coords. Keep until then as a backstop; remove in the cleanup commit at the end of V10.8.C.
